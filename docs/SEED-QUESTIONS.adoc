= Spline — Seed Questions
:doctype: book
:revdate: 2026-06-16
:toc: macro
:toclevels: 2
:sectnums:

[abstract]
spline is the least-settled layer of the cleave/groove/spline trio and the spot where
`protocol-squisher` originally planned to operate and stalled. This document is the
seed: scope, non-goals, the squisher caution, and the open questions to resolve
*before* any implementation. There is deliberately NO lifted code here.

toc::[]

== Why this is seed-only

The recommendation was to defer the data-form *inside groove* (least-settled,
squisher-adjacent). The owner chose to spin it off as its own repo now. To honour both
— the decision and the caution — spline exists as a repo but stays thin until the
questions below are answered. Revert cost is therefore low (delete a local dir, or fold
the docs back into groove).

== Scope (what spline IS)

* The *serialise* + *parallelise* data-form: the bytes Groove carries.
* A thin *alignment* over existing transports, NOT a new serialiser:
  - control plane = Bebop (typed, binary, reliable/ordered)
  - media plane = WebRTC/RTP (lossy, low-latency)
* A *typed boundary* for groove capability tokens + handles (vs bare scalars / opaque
  JSON).

== Non-goals (what spline IS-NOT)

* IS-NOT the interface engine (that is `cleave`).
* IS-NOT the protocol / negotiation (that is `groove`).
* IS-NOT a new wire format invented from scratch (that is the squisher trap).
* IS-NOT posture-aware: by B perp C, spline stays blind to the cleave membrane's
  lifecycle/security witness. The rank/lifecycle witness MUST NOT appear in the byte
  encoding.

== The squisher caution (load-bearing)

`protocol-squisher` is a *frozen provenance archive*; its v1.x universal-interop claim
was closed by `hub_ceiling` (transport composition = max/bottleneck). spline must:

. NOT lift code from `protocol-squisher` / `squisher-corpus`.
. NOT recreate the design-source -> copy/sync seam (cleave is a *library groove depends
  on*, not a thing copied from; the same discipline applies to spline vs its consumers).
. Treat the squisher archive as *evidence of what stalled*, not a foundation.

== Open questions (resolve before implementing)

The original question text is kept below for provenance; each is annotated with
its resolution.

. **Name.** Is `spline` right, or should this fold back into `groove` as a module? (Cheap
  to change now — local-only.)
+
*RESOLVED 2026-07-06* — standalone repo confirmed, named `spline` (renamed from
the working name `weft`). See the joinery-naming ADR: groove ADR 0009
(`groove/docs/decisions/0009-the-joinery-naming-canon.adoc`).
. **"Parallelise" — what does it mean here?** The name says serialise *and* parallelise.
  Is "parallelise" about (a) multiplexing independent streams over one connection, (b)
  fan-out/scatter of one logical message, (c) parallel (de)serialisation for throughput,
  or (d) something else? This is undefined and must be pinned.
+
*RESOLVED BY ADR* — xref:decisions/0002-parallelise-means-plane-multiplexing.adoc[ADR 0002]:
reading (a), sharpened to *typed planes* of one groove session, multiplexed
without head-of-line blocking; (b) and (c) explicitly rejected.
. **Bebop binding.** How is the Bebop schema authored/owned — in spline, or referenced from
  the consumer (burble) that already has it? (Avoid duplicating burble's schema.
  Correction 2026-07-09: burble is MPL-2.0, owner-sole, and editable; the earlier
  son-shared-AGPL note was wrong — AGPL-3.0-or-later applies to son-shared repos such as
  idaptik, not burble. Schema ownership is decided in ADR 0003.)
+
*RESOLVED BY ADR* — xref:decisions/0003-bebop-schema-ownership.adoc[ADR 0003]:
burble's `.bop` schemas are canonical for the messages burble owns; spline owns
only the alignment mapping (never a copied schema — that is the squisher seam).
. **Typed tokens.** What exactly is the typed boundary for groove cap tokens + handles,
  and how does it relate to cleave's linear-handle newtype (PROOF-NEEDS G-3) and manifest
  signing (G-5)?
+
*RESOLVED BY ADR* — xref:decisions/0004-typed-cap-token-boundary.adoc[ADR 0004]:
tokens = signed manifest capability records (groove ADR 0010); handles =
wire-opaque strings mapped to linear handles at each endpoint; spline carries
the opaque forms and stays posture-blind (TS-7).
. **Promotion criteria.** What has to be true for spline to be considered "stable"? Until
  then it stays seed/thin.
+
*RESOLVED BY ADR* — xref:decisions/0005-promotion-criteria.adoc[ADR 0005]: four
criteria to leave seed (live Bebop wiring in burble, second consumer, mapping
conformance test in groove CI, typed-boundary pairing demo), plus what "stable"
additionally needs.

== Status

Still seed. No implementation, and no `src/` until the ADR 0005 bar is met —
that ADR is the promotion bar. The five open questions are answered *on paper*
(Q1 resolved 2026-07-06; Q2–Q5 resolved by ADRs 0002–0005, 2026-07-09); none of
the answers is wired yet. Captured 2026-06-16 so the decision (spin-off now,
kept thin) and the caution are explicit and not lost across compaction.
