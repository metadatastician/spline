---

== What it is

`weft` is the *serialise + parallelise* data-form wire layer — the bytes that Groove
carries. The textile metaphor pairs with the estate's `gossamer` and `groove`: the
*weft* is the crosswise thread woven through the warp.

It is **not a new serialiser.** It is a thin *alignment* over what consumers already
speak:

[cols="1,3"]
|===
| Control plane | **Bebop** — typed, binary, reliable/ordered. Gives Groove's logical schema a real typed encoding.
| Media plane | **WebRTC / RTP** — lossy, low-latency, no head-of-line block.
|===

(The 2026-06-15 gossamer audit found `burble` already speaks both — so weft *carries*
those typed planes rather than inventing a format.)

== The squisher caution (why this repo is thin)

The data-form is **exactly where `protocol-squisher` originally planned to operate and
stalled.** weft is a *fresh start*: it must NOT recreate the copy/sync seam that stalled
squisher, and must NOT lift code from `protocol-squisher` / `squisher-corpus` (a frozen
provenance archive — evidence, not a source).

This repo is therefore intentionally seed-only for now: scope, non-goals, and open
questions, no lifted implementation. See `docs/SEED-QUESTIONS.adoc`.

== Where it sits

`weft` is consumed by `groove` alongside `cleave`. The *B perp C* decoupling means weft
stays **posture-blind**: the cleave membrane contract (lifecycle, permeability, security)
is independent of the wire format, and the rank/lifecycle witness MUST NOT appear in
weft's byte encoding.

== Open naming/scoping question

Whether this stays a standalone repo named `weft`, or folds back into `groove` as a
module, is still open and cheap to change (local-only). Recorded so the decision is
explicit rather than drifting.
